## Observations

**OBS-001 ‚Äî Repeat passes each create a separate session log**

When `xN` repeat is used (top-level or inside a composition branch), each pass through `executeWork` creates its own session log file. For `cook "work" x3`, this produces 3 separate `.cook/logs/` files rather than one shared log.

Confirmed in T18: the `x2` branch inside a `vs` composition produced 2 log files, one per repeat pass.

The research doc states "Session log continuity ‚Äî all steps in a single cook execution share one session log." That constraint was written in the context of the review loop (work‚Üíreview‚Üígate). Whether it extends to repeat passes is not explicitly addressed. The current behaviour is consistent (each executeWork is a discrete call that creates its own log), but diverges from the principle if a user expects a single log to span an entire `x3 repeat` execution.

**Not a blocking issue** ‚Äî logs are not lost, and each pass is individually traceable. Worth a design decision before closing this PR: should `executeRepeat` pre-create a shared log file and thread it through pass contexts, or is per-pass logging acceptable?

**OBS-002 ‚Äî `compare v3` silently drops `v3`**

`cook "A" vs "B" compare v3` runs as a normal 2-branch compare; the `v3` token is silently ignored. No parser error emitted. This is tracked in TODO.md.
